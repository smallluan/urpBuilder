# 什么是 BuilderNext
BuilderNext 是面向**可视化搭建、数据资源接入与前端工程化交付**打造的一体化工作台平台。

其设计目标并非生成可预览的静态页面，而是将企业级业务中高频复用的前端生产工作，统一收敛至同一体系内：
- 可视化搭建页面
- 设计与管理可复用组件
- 管理常量、数据表、云函数、素材等运行时资源
- 支持个人空间/团队空间的资源上下文切换
- 通过统一的预览、发布、版本对比机制，完成从草稿到可交付产物的完整链路

一句话概括：**BuilderNext 是整合页面搭建、组件开发、数据接入、运行预览、版本管控于一体的全链路前端搭建平台。**

<DocsCallout title="定位说明">
BuilderNext 本质是一套面向团队内部的工程化实施方案，在此基础上封装为可视化界面。平台各项能力并非孤立设计，而是围绕「稳定、可持续地生产页面与组件资产」这一核心目标进行组织。
</DocsCallout>

## 核心解决问题
传统内部搭建平台普遍存在流程碎片化问题：
- 仅支持画布拖拽编排，数据接入能力薄弱
- 支持接口配置，但页面、组件、资源相互割裂
- 具备预览能力，但缺乏版本管理机制
- 支持团队协作，但资源边界模糊，长期难以治理

BuilderNext 旨在解决**端到端完整链路问题**：
> 在单一平台内，完成页面/组件构建、资源组织、数据接入、调试预览、版本发布与团队协作的全流程工作。

从现有架构与功能体系来看，BuilderNext 已围绕该目标形成完整的能力版图。

## 使用者核心能力
### 1. 页面搭建
BuilderNext 将页面定义为可管理、可运维的页面模板，而非孤立画布：
- 创建页面并维护基础元信息
- 在编辑器中构建页面结构与组件树
- 配置路由信息（routePath、pageTitle、menuTitle 等）
- 绑定页面发布与预览所依赖的组件库
- 实时预览页面真实运行时效果
- 草稿提交与正式版本发布

平台不仅关注组件布局编排，同时完整覆盖**页面访问、预览、发布、上线**全生命周期。

### 2. 组件搭建
BuilderNext 将**自定义组件**定位为一等公民资源：
- 创建独立可维护的组件模板
- 在专属编辑器中配置组件结构、属性与交互逻辑
- 支持组件独立预览与调试
- 组件版本发布与历史管理
- 多版本可视化 Diff，对比 UI、结构与代码差异

核心价值在于：平台不只是页面拼装工具，更是**可复用组件资产的沉淀与管理体系**。

### 3. 运行时数据资源管理
BuilderNext 对数据资源进行精细化分类，而非单一 API 抽象：
- **常量**：管理字符串、数字、对象、数组等静态配置
- **数据 API**：管理 HTTP 接口配置（能力持续完善中）
- **云函数**：承载服务端逻辑，支持部署与状态管理
- **数据表**：与云函数协同构成平台数据操作基础能力
- **素材库**：按目录结构管理图片、文件等静态资源

设计理念：**页面呈现是最终结果，资源体系的统一性与规范性决定前端交付效率与质量。**

### 4. 预览与运行时验证
BuilderNext 内置独立预览引擎，而非静态截图式预览：
- 根据页面/组件 ID 远程加载模板实时渲染
- 支持页面预览、组件预览双模式
- 支持路由路径级站点预览
- 支持 RouteOutlet 路由组合渲染
- 内置运行时、数据中枢、调试面板等调试能力

预览并非最终校验环节，而是**开发过程中运行时验证的核心能力**。

### 5. 版本、发布与回溯
BuilderNext 严格区分**草稿保存**与**版本发布**：
- 草稿保存不生成版本
- 发布操作生成正式可追溯版本
- 页面发布自动关联预览组件库
- 组件支持版本列表与历史回溯
- 支持多版本可视化差异对比
- 支持历史版本回滚为新草稿基线

该机制明确区分：
- 临时编辑草稿（非正式）
- 可被团队引用与交付的正式版本快照

### 6. 团队协作与空间隔离
BuilderNext 采用**个人空间 / 团队空间**的双层资源隔离模型：
- 个人空间：管理私有资源，独立试验与开发
- 团队空间：管理团队共享资源，协同生产
- 页面、组件、常量、素材等资源随空间自动切换上下文
- 支持成员、管理员、所有者等多级权限控制

设计原则：**资源边界清晰是协作可控的前提**，在此基础上构建权限、归属、共享体系。

## 平台定位：体系化平台而非单点工具
从单一功能视角，BuilderNext 可能被误认为：
- 低代码编辑器
- 组件管理平台
- 资源配置后台
- 文档站点
- 预览服务

从整体架构看，BuilderNext 是**覆盖前端生产全流程的体系化平台**，核心分层如下：

| 层级 | 核心能力 |
| --- | --- |
| 搭建层 | 页面编辑、组件编辑、画布编排、属性配置 |
| 资源层 | 常量、数据表、云函数、素材、API 配置 |
| 运行层 | 预览引擎、流程运行时、数据中枢、站点预览 |
| 发布层 | 草稿管理、版本发布、组件库绑定、版本历史 |
| 协作层 | 个人/团队空间、团队看板、权限管控、贡献管理 |

因此其定位是**BuilderNext 平台**，而非单一页面编辑器。

## 命名释义：BuilderNext
- **Builder**：直观体现核心形态——可视化搭建平台。
- **Next** 包含两层核心含义：
  1. **下一代工作模式**：重构分散的页面、组件、数据、素材、发布流程，形成一体化工作流。
  2. **可持续演进架构**：现有能力为基础底座，未来可向流程编排、资产治理、质量管控等方向持续扩展。

名称旨在表明：平台并非一次性实验工具，而是**可长期迭代、持续演进的工程化搭建体系**。

## 适用场景
BuilderNext 更适用于以下场景：
- 团队内部中后台系统、运营页面、业务组件的持续交付
- 组件资产化、标准化沉淀与复用
- 页面与配置、数据、素材、服务逻辑统一治理
- 需要标准化草稿、预览、发布、版本对比流程
- 需要在个人开发与团队协作之间灵活切换上下文

平台并非面向全用户的零门槛通用产品，而是**具备明确工程规范的前端专业化生产平台**。

## 平台核心认知
使用 BuilderNext 的核心前提：
> BuilderNext 的核心单元不是「页面快照」，而是**可长期维护、可治理、可复用的前端资产**。

在此视角下，页面、组件、常量、云函数、素材、预览、发布、版本等功能并非孤立存在：
- 页面依赖组件
- 组件依赖数据与素材
- 数据与逻辑依赖运行时
- 运行效果依赖预览验证
- 交付产物依赖版本管控
- 团队协作依赖资源隔离

BuilderNext 的核心价值，是将上述天然关联的链路**完整产品化、平台化**。
